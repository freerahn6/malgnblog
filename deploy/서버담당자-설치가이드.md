# 블로그 자동 갱신 설치 요청 (서버 담당자용)

`blog.malgnsoft.com` 정적 블로그를 서버에 올리고, 이후 내용이 바뀌면 자동으로 갱신되게 하는 작업입니다.
**최초 1회 설정 이후에는 담당자가 손댈 일이 없습니다.**

---

## 1. 요약 — 서버에서 무슨 일이 벌어지나

2분마다 `git`으로 GitHub의 정적 HTML을 내려받아, 바뀐 게 있으면 웹 디렉터리에 반영합니다.

```
서버 cron (2분마다)
   └─ git fetch  ──아웃바운드 HTTPS──▶ github.com (공개 저장소)
   └─ rsync      ──▶ 웹 디렉터리 갱신
```

**보안 관점 요약**

| 항목 | 내용 |
|---|---|
| 인바운드 포트 개방 | **없음** |
| 서버에 저장하는 자격증명·키·토큰 | **없음** (공개 저장소라 인증 불필요) |
| 외부에 제공하는 계정 | **없음** |
| 통신 방향 | 서버 → `github.com:443` **아웃바운드 전용** |
| 서버에서 실행되는 프로그램 | `git`, `rsync` **둘뿐** (저장소의 코드를 실행하지 않음) |
| 설치하는 런타임·패키지 | **없음** (Python·Node·DB 불필요) |
| 받아오는 내용 | 정적 HTML·XML·TXT뿐. 실행 파일·스크립트 없음 |

> 받아오는 파일은 전부 정적 문서이고, 서버는 그것을 **서빙만** 합니다.
> 저장소에서 받은 스크립트를 실행하는 동작은 어디에도 없습니다.

**저장소:** `https://github.com/freerahn6/malgnblog` (공개) — 브랜치 `deploy`

---

## 2. 사전 확인

```bash
git --version && rsync --version | head -1     # 없으면 설치 필요
curl -sI https://github.com | head -1          # HTTP/2 200 이면 아웃바운드 통신 정상
```

방화벽에서 아웃바운드 443이 막혀 있다면 `github.com` 만 허용해주시면 됩니다.

---

## 3. 설치 (최초 1회)

아래 두 값을 환경에 맞게 정한 뒤 진행해주세요.

- `WEBROOT` — 블로그를 서빙할 웹 디렉터리 (예: `/var/www/blog`)
- 작업 계정 — 아래 예시는 전용 계정 `blogsync`를 만들어 씁니다(웹루트만 쓰는 최소 권한)

### 3-1. 전용 계정과 작업 디렉터리

```bash
sudo useradd -r -s /usr/sbin/nologin -d /srv/malgnblog blogsync
sudo mkdir -p /srv/malgnblog /var/www/blog
sudo chown -R blogsync:blogsync /srv/malgnblog /var/www/blog
```

### 3-2. 최초 내려받기 + 첫 반영

```bash
sudo -u blogsync git clone --branch deploy --single-branch --depth 1 \
     https://github.com/freerahn6/malgnblog.git /srv/malgnblog/site

sudo -u blogsync rsync -a --delete --exclude .git \
     /srv/malgnblog/site/ /var/www/blog/

ls /var/www/blog          # index.html, sitemap.xml, lms/, hrd/ ... 가 보이면 성공
```

> `.git` 디렉터리는 `--exclude` 로 웹루트에 복사되지 않습니다(저장소 메타데이터 노출 방지).

### 3-3. 자동 갱신 등록

```bash
sudo tee /etc/cron.d/malgnblog >/dev/null <<'EOF'
# 맑은소프트 블로그 자동 갱신 — 2분마다 변경분 확인 후 반영
SHELL=/bin/bash
PATH=/usr/local/bin:/usr/bin:/bin
*/2 * * * * blogsync cd /srv/malgnblog/site && git fetch -q origin deploy && git reset -q --hard origin/deploy && rsync -a --delete --exclude .git /srv/malgnblog/site/ /var/www/blog/
EOF
sudo chmod 644 /etc/cron.d/malgnblog
```

변경이 없으면 `git fetch`가 수 KB만 주고받고 끝납니다(부하 무시할 수준).

### 3-4. 웹서버 설정

`WEBROOT`를 서빙하도록 설정해주세요. nginx 예시는 동봉된 `nginx-blog.conf`에 있습니다.
핵심은 두 가지입니다.

```nginx
root /var/www/blog;
index index.html;

location / {
    try_files $uri $uri/ $uri/index.html =404;
}

# 조회수 수집 요청. 서버에 해당 기능이 없어 404 로그만 쌓이므로 조용히 받아넘깁니다.
location = /api/track { return 204; }
```

SSL은 Let's Encrypt 등 기존 운영 방식대로 발급해주시면 됩니다.

## 4. 동작 확인

```bash
# cron이 도는지
grep malgnblog /var/log/syslog | tail -3        # 또는 journalctl -u cron | tail

# 현재 반영된 버전
sudo -u blogsync git -C /srv/malgnblog/site log -1 --format='%h %ci %s'

# 수동으로 즉시 갱신
sudo -u blogsync bash -c 'cd /srv/malgnblog/site && git fetch -q origin deploy && git reset -q --hard origin/deploy && rsync -a --delete --exclude .git /srv/malgnblog/site/ /var/www/blog/'
```

---

## 5. 중단·제거

```bash
sudo rm /etc/cron.d/malgnblog          # 자동 갱신만 중단(사이트는 그대로 유지)
sudo rm -rf /srv/malgnblog             # 작업 디렉터리까지 제거
sudo userdel blogsync
```

웹루트(`/var/www/blog`)의 파일은 그대로 남으므로 사이트는 계속 서비스됩니다.

---

## 6. 참고

- 콘텐츠 담당자가 글을 발행하면 GitHub에서 자동으로 빌드·검증된 뒤 `deploy` 브랜치에 올라갑니다.
  **검증에 실패한 결과물은 `deploy` 브랜치에 올라가지 않으므로, 서버에도 반영되지 않습니다.**
- 서버가 받는 것은 항상 검증을 통과한 완성본입니다.
